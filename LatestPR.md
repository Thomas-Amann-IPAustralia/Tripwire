## Summary

This PR implements the final three stages of the Tripwire monitoring pipeline (Stages 7–9) along with supporting infrastructure for feedback ingestion and pipeline orchestration.

## Key Changes

### Core Pipeline Implementation
- **`src/pipeline.py`** (855 lines): Main pipeline orchestrator that coordinates Stages 1–9. Handles:
  - Configuration loading and SQLite database initialization
  - Processing of pending deferred triggers from previous runs
  - Source registry loading and per-source processing through Stages 1–6
  - Trigger aggregation (Stage 7), LLM assessment (Stage 8), and notification (Stage 9)
  - Git commit of updated snapshots and database state
  - GitHub Actions Job Summary generation
  - Observation mode support (skips Stages 8–9 during calibration)

### Stage 7: Trigger Aggregation
- **`src/stage7_aggregation.py`** (305 lines): Groups triggers by IPFR page before LLM assessment
  - `TriggerSource`: represents one upstream change event
  - `TriggerBundle`: groups all triggers for a single IPFR page
  - `aggregate_triggers()`: consolidates (source, page) pairs to prevent duplicate notifications
  - Preserves all scoring metadata from Stages 4–6 for LLM context

### Stage 8: LLM Assessment
- **`src/stage8_llm.py`** (743 lines): Makes single LLM call per IPFR page to determine amendment necessity
  - Comprehensive system prompt guiding the LLM to assess whether external changes require IPFR updates
  - JSON schema validation with retry logic (up to 2 attempts per bundle)
  - Three verdict types: `CHANGE_REQUIRED`, `NO_CHANGE`, `UNCERTAIN`
  - Deferred trigger mechanism: failed assessments are written to SQLite for retry in next run
  - Observation mode: skips LLM calls entirely to reduce costs during calibration
  - Token usage tracking for cost monitoring

### Stage 9: Notification
- **`src/stage9_notification.py`** (748 lines): Sends consolidated email to content owner
  - Composes HTML and plain-text email with:
    - One section per flagged IPFR page
    - Source URLs and normalized diffs that triggered the alert
    - LLM reasoning and suggested amendments
    - Scoring evidence (relevance, bi-encoder, cross-encoder scores)
    - Four mailto feedback links per page (useful, not_significant, wrong_amendment, wrong_page)
  - Separate sections for UNCERTAIN verdicts and rejected candidates
  - SMTP delivery with retry logic and fallback file saving on failure
  - No-alert policy: email not sent if no pages flagged

### Feedback Ingestion
- **`src/feedback_ingestion.py`** (362 lines): Gmail IMAP polling for feedback replies
  - Parses structured feedback emails generated by Stage 9 mailto links
  - Extracts run_id, page_id, source_id, and feedback category
  - Appends records to `data/logs/feedback.jsonl`
  - Marks successfully parsed messages as read; leaves malformed messages for retry
  - Uses only Python standard library (imaplib, email, json, re)

### Workflow Updates
- **`.github/workflows/tripwire.yml`**: Updated main pipeline workflow
  - Runs daily at 02:00 UTC (after ingestion at 01:00 UTC)
  - Concurrency control to prevent simultaneous writes to SQLite
  - Configurable run_id override and log level via workflow dispatch
  - 30-minute timeout per Section 6.6
  - Commits updated snapshots and database back to repository

- **`.github/workflows/feedback_ingestion.yml`** (new): Dedicated feedback ingestion workflow
  - Runs every 6 hours
  - Polls Gmail for feedback replies
  - Commits updated feedback.jsonl to repository
